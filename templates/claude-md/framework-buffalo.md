## Buffalo Conventions

### Project Structure
- Follow Buffalo scaffold: `actions/`, `models/`, `templates/`, `locales/`, `migrations/`
- Entry point: `main.go` with `app.Serve()` (generated by `buffalo new`)
- Actions: `actions/{resource}.go` for HTTP handlers
- Models: `models/` using Pop ORM for database interaction
- Templates: `templates/` with Plush templating engine
- Migrations: `migrations/` for database schema changes via Pop
- Assets: `assets/` compiled with Webpack

### Action Patterns
- Actions receive `buffalo.Context`: `func UsersIndex(c buffalo.Context) error`
- Resource generator: `buffalo.Resource` interface for RESTful CRUD (List, Show, Create, Update, Destroy)
- Register resources: `app.Resource("/users", UsersResource{})` for full RESTful routes
- Access params: `c.Param("id")`; bind body: `c.Bind(&user)`
- Respond: `c.Render(200, r.JSON(data))` or `c.Render(200, r.HTML("users/index.plush.html"))`

### Pop (ORM) Patterns
- Define models with `pop.Model` and struct tags: `db:"name" json:"name"`
- Migrations: `buffalo pop generate fizz create_users` for Fizz-format migrations
- Queries: `models.DB.Find(&user, id)`, `models.DB.Where("active = ?", true).All(&users)`
- Eager loading: `models.DB.Eager("Posts").Find(&user, id)` to avoid N+1
- Validations: implement `Validate` method on models returning `*validate.Errors`

### Middleware
- Register in `actions/app.go` within the `App()` function
- Built-in: `csrf.New`, `i18n.New`, `paramlogger.ParameterLogger`
- Auth: use `buffalo-pop/pop/popmw` for transaction middleware
- Group middleware: `api := app.Group("/api"); api.Use(authMiddleware)`
- Database transaction middleware: wraps each request in a DB transaction

### Testing
- Test actions with `httptest.NewRequest` and the generated test suite
- Use `suite.NewAction(app)` for action tests with full app context
- Model tests: `suite.NewModel()` with test database
- Fixtures: load test data from `fixtures/` directory
- Database: tests use a separate test database, rolled back between tests

### Anti-Patterns to Avoid
- Do not skip Pop migrations — manual schema changes cause drift with the codebase
- Do not use `models.DB` directly in actions — use the transaction from context `c.Value("tx").(*pop.Connection)`
- Do not put business logic in actions — extract to service functions
- Do not skip model validations — they run automatically on save but document expectations
- Do not use Plush templates for API-only applications — use `r.JSON()` renderer
